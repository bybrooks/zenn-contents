DB のレコードに対して、データの削除を表現する方法として、おおきく「物理削除」と「論理削除」がある。
さらに論理削除では、削除を表現するいくつかの方法がある。
これらのメリデメとそれぞれの利用シーンを整理する。

# 物理削除

SQL の `DELETE` 文を使って、DBのディスク上からデータを完全に削除する方法。

#### メリット

- 容量を圧迫せず、DBを軽量に保てる
- インデックスのサイズが抑えられ、検索パフォーマンスが良い
- 一意性制約（Unique Key）のバグが起きない
- `WHERE` 句で削除フラグを考慮する必要がない

#### デメリット

- 誤って消した場合の復旧が非常に困難（バックアップから戻す必要あり）
- 「いつ、誰が消したか」という変更履歴・監査ログが残らない
- 外部キー制約（Foreign Key）がある場合、関連データも消すかエラーになる

#### 主なユースケース

- 一時的なセッションデータ、キャッシュデータ
- プライバシー保護（GDPR等）の観点で、ユーザーから完全に消去を求められる個人情報
- 再作成が容易な中間データ

# 論理削除

データは物理的に削除せず、削除されたかどうかを表すフラグカラムを使用して削除を表現する方法。
論理削除はいくつかの実現方法がある。

## 論理削除 - フラグ型

`is_deleted` など、削除されたかどうかを 真偽値 で表現したカラムを利用するパターン。

#### メリット

- `UPDATE` で済むため、フラグを戻すだけで簡単にデータを復旧できる
- 他テーブルからの外部キー制約を壊さない

#### デメリット

- すべての検索クエリに `WHERE is_deleted = false` を含める必要があり、バグの温床になりやすい
- 一意性制約（Unique Key）との相性が最悪（例：削除済みユーザーと同じメールアドレスで再登録できない）
- データが溜まり続けるため、パフォーマンスが低下する恐れがある

#### 主なユースケース

- 誤操作による削除の防止・復旧が強く求められるマスタデータ（商品マスタ、ユーザーマスタなど）

## 論理削除 - 日時型 

フラグの代わりに「削除された日時」を記録するパターン。
多くのモダンなWebフレームワーク（Ruby on Railsのparanoiaなど）で標準採用されている。

- **テーブル定義例**: `deleted_at` (DATETIME / TIMESTAMP、初期値は `NULL`)
- **実装例**: `UPDATE users SET deleted_at = NOW() WHERE id = 1;`
- **参照時**: `SELECT * FROM users WHERE deleted_at IS NULL;`
    

#### メリット

- フラグ型のメリットに加え、「いつ削除されたか」という監査情報が同時に残る
- DBによっては、一意性制約の回避策がある（PostgreSQLの部分インデックス機能などを使えば、`deleted_at IS NULL` のレコード間だけでUniqueを保証できる）

#### デメリット

- フラグ型同様、クエリの複雑化やデータ肥大化の問題は残る

#### 主なユースケース

- 現在のWebアプリケーション開発における、一般的な論理削除のデファクトスタンダード

## 論理削除 - ステータス管理型

削除を単独の事象として捉えるのではなく、**データの「ライフサイクル（状態）」の一部として管理**するパターンです。

- **テーブル定義例**: `status` (VARCHAR や ENUM)
- **状態の例**: `'draft'` (下書き) → `'active'` (公開中) → `'suspended'` (一時停止) → `'deleted'` (削除)

#### メリット

- データの「状態遷移」を一つのカラムで美しく表現できる
- ビジネスロジックとデータベースの構造が一致しやすい

#### デメリット

- 「有効なデータ」を取得する条件が、`status = 'active'` だけでなく `'suspended'` も含む場合など、ビジネスロジックの変更に応じてクエリが複雑化しやすい

#### 主なユースケース

- ブログやECサイトの記事・商品管理（下書き、公開、非公開、削除）
- 会員のステータス管理（仮登録、本登録、退会、凍結）

## 論理削除 - 履歴・アーカイブテーブル移動型

削除するデータを、**メインテーブルから物理削除し、同時に「履歴専用（またはアーカイブ専用）の別テーブル」へ移動**させるパターンです。

#### メリット

- メインテーブルが常に軽量に保たれるため、検索パフォーマンスが落ちない
- 一意性制約の問題がまったく起きない
- アーカイブ側で過去データの分析や監査ができる

#### デメリット

- テーブルが2つになるため、スキーマ変更（カラム追加など）の際に両方のテーブルを修正する必要があり、メンテナンスコストが上がる
- 移動処理をトランザクションで厳密に制御する必要がある

#### 主なユースケース

- 注文履歴、決済データ、配送データなど、完了した後は基本的に変更されないが、削除せず残す必要がある巨大なデータ

## 論理削除 - 有効期間管理型

データに「有効開始日時」と「有効終了日時」を持たせ、時間軸でデータを制御するパターンです。

- **テーブル定義例**: `valid_from`（開始日） と `valid_to`（終了日、未来永劫を表す `9999-12-31` などが入る）
- **削除の実装例**: `UPDATE products SET valid_to = NOW() WHERE id = 1;`
- **参照時**: `SELECT * FROM products WHERE NOW() BETWEEN valid_from AND valid_to;`

#### メリット

- 「過去の特定の時点でどうだったか」を完全に再現できる（タイムトラベルクエリ）
- 「来週から適用する価格」のような、未来のデータも事前に登録できる

#### デメリット

- 設計とクエリ（特に期間が重複しないようにする制約など）が非常に複雑になる
- 開発者の学習コストが高い


#### 主なユースケース

- 商品の価格改定履歴、組織変更に伴う所属の履歴、契約情報

| **パターン**         | **復旧の難易度**      | **検索性能への影響** | **Unique制約の扱い** | **おすすめ度・主な用途**     |
| ---------------- | --------------- | ------------ | --------------- | ------------------ |
| **1. 物理削除**      | 🛑 不可能（要バックアップ） | 🟢 非常に良い     | 🟢 影響なし         | 個人情報や一時的なデータ       |
| **2. 論理削除（フラグ）** | 🟢 簡単           | 🟡 やや悪化      | 🛑 困難           | 単純なシステム（日時が不要な場合）  |
| **3. 論理削除（日時）**  | 🟢 簡単           | 🟡 やや悪化      | 🔺 工夫が必要        | **Web開発のデフォルト推奨**  |
| **4. ステータス管理**   | 🟢 簡単           | 🟡 やや悪化      | 🛑 困難           | 状態遷移が多いビジネスロジック    |
| **5. アーカイブ移動**   | 🟡 やや面倒         | 🟢 非常に良い     | 🟢 影響なし         | **大量のトランザクションデータ** |
| **6. 有効期間管理**    | 🟢 簡単           | 🔺 インデックス工夫要 | 🛑 非常に困難        | 履歴や未来の予定を厳密に管理する場合 |
